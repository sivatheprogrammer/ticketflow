# ADR-017: AKS Deployment Challenges and Lessons Learned

## Status
Accepted

## Context

Phase 6 Step 2 added an AKS + Helm demonstration track alongside the primary
Azure Container Apps (ACA) deployment (see ADR-015, ADR-016). While ACA
deployment was comparatively straightforward, standing up the same three
services (`ticketflow-api`, `ticketflow-identity`, `ticketflow-gateway`) on
raw Kubernetes surfaced a series of non-obvious issues. This ADR documents
those issues, root causes, and fixes, since several of them are classic AKS
gotchas worth carrying into future Kubernetes work.

## Challenges Encountered

### 1. VM SKU / quota mismatch for the node pool

**Symptom:** `terraform apply` failed provisioning the AKS node pool with
`Standard_B2s` — "VM size not allowed in your subscription in this location."

**Root cause:** Two independent constraints had to both be satisfied:
- SKU availability in the region (`az vm list-skus`)
- Subscription vCPU quota per VM family (`az vm list-usage`)

`Standard_B2s` (non-v2) wasn't orderable in `centralus` at all — Azure
returned `Standard_B2s_v2` instead when queried. `Standard_B2s_v2` *was*
orderable, but the subscription had **0 quota** in the `Standard Bsv2 Family`.
Meanwhile the older `Standard BS Family` quota (10 vCPUs) was unusable
because no SKU in that family was actually orderable in the region.

**Fix:** Switched to `Standard_D2s_v3` — orderable in `centralus`, with
10 vCPU quota available under `Standard DSv3 Family`. General lesson:
region availability and subscription quota must be checked independently
before picking a VM size; neither check alone is sufficient.

### 2. Container Apps failed to redeploy after `terraform destroy` / re-`apply`

**Symptom:** After destroying and recreating all infrastructure (new empty
ACR), Container Apps failed with `MANIFEST_UNKNOWN: manifest tagged by
"latest" is not found`.

**Root cause:** `terraform destroy` removes the ACR along with its images.
A fresh `terraform apply` recreates an empty registry, but the Container App
definitions still reference `:latest` tags that no longer exist until images
are rebuilt and pushed.

**Fix:** Rebuilt and pushed all three images to the new ACR before
re-applying. General lesson: image builds are not part of Terraform's
lifecycle — they must be sequenced manually (or via CI/CD) after any
ACR-destroying operation, before dependent compute resources are applied.

### 3. Terraform state desync after partial resource creation

**Symptom:** Re-running `terraform apply` after the ACR was populated failed
with "a resource with this ID already exists — needs to be imported."

**Root cause:** The first `apply` attempt had actually created the Container
Apps in Azure, but they entered `Failed` provisioning state (due to #2).
Terraform did not persist them to state on that failed run, so Terraform's
state and Azure's actual resources diverged.

**Fix:** `terraform import` was attempted but also failed, since Azure
refuses to return full resource details (including secrets) for a resource
stuck in `Failed` state. The pragmatic fix was to delete the failed Container
Apps directly via `az containerapp delete` and let Terraform create them
fresh. General lesson: a failed apply can leave real cloud resources without
corresponding Terraform state; when import also fails due to the resource's
own broken state, deleting and recreating is often faster than fighting the
state file.

### 4. Helm release naming produced double-suffixed resource names

**Symptom:** Pods failed with `CreateContainerConfigError`, referencing a
Secret name that didn't match what was actually created
(`ticketflow-api-api-secrets` expected vs. `ticketflow-api-secrets` created).

**Root cause:** Each chart's templates build resource names as
`{{ .Release.Name }}-api-secrets` (or `-identity-secrets`, `-gateway-secrets`).
Since each chart's Helm release was itself named `ticketflow-api` /
`ticketflow-identity` / `ticketflow-gateway`, the release name already
contained the service name, and the template suffix duplicated it —
producing `ticketflow-api-api`, `ticketflow-identity-identity`, etc. This
also applied to Service names generated by the same pattern.

**Fix:** Recreated Secrets under the actual (doubled) names Kubernetes
expected, and updated cross-service references (e.g. Gateway's proxy target
URLs) to point at the real Service names. General lesson: when a Helm chart
name and its intended release name overlap, template-generated resource
names should avoid re-appending the chart's own name — otherwise naming
collisions like this are easy to introduce and easy to miss until a pod
actually fails to start.

### 5. Connection string key names didn't match `appsettings.json`

**Symptom:** Both API and Identity pods threw
`PlatformNotSupportedException: LocalDB is not supported on this platform`
immediately at startup, despite Secrets being correctly created and wired.

**Root cause:** The Helm chart's `secretEnv` keys were guessed
(`ConnectionStrings__Default`, `ConnectionStrings__DefaultConnection`)
rather than matched against the actual key names in each service's
`appsettings.json`. .NET's configuration system silently ignores environment
variables that don't match an existing configuration path — it does not
error on an unrecognized override, it simply falls back to whatever default
is baked into `appsettings.json` (LocalDB, in this case).

**Fix:** Inspected each service's `appsettings.json` directly and matched
the Secret/env var key exactly: `ConnectionStrings__Default` for the main
API, `ConnectionStrings__IdentityDb` for Identity (a different key name,
since Identity uses its own `IdentityDbContext`). General lesson: .NET
config overrides via environment variables fail silently on a name
mismatch — always verify against the actual `appsettings.json`, never guess
the key name from convention alone.

### 6. Shared database caused a migration collision between services

**Symptom:** After fixing the connection string key, Identity's EF Core
migration failed with `SqlException: There is already an object named
'Customers' in the database.`

**Root cause:** Both `TicketFlow.Api` and `TicketFlow.Identity.Api` were
pointed at the same Azure SQL database (`sqldb-ticketflow-dev`). Each
service has its own `DbContext` and its own migrations, but since they
shared one physical database, their independently-run migrations collided
on an overlapping table name.

**Fix:** Provisioned a second Azure SQL database
(`sqldb-ticketflow-identity-dev`) on the same logical server, dedicated to
the Identity service, and repointed its connection string accordingly. This
aligns with the "each service owns its own data" principle already
established when Identity was extracted from the modular monolith in Phase
5 — sharing a physical database undermines that boundary even if each
service has a distinct `DbContext`.

### 7. YARP reverse proxy configuration path mismatch

**Symptom:** Gateway pod responded `502 Bad Gateway`; its logs showed YARP
attempting to proxy to `https://localhost:55214` — a hardcoded local
development port — rather than the Kubernetes Service the Gateway was
supposed to route to.

**Root cause:** The Helm chart's environment variable names for the YARP
`ReverseProxy` configuration did not match the actual hierarchy in the
Gateway's `appsettings.json`. The real structure nests cluster ID and
destination name as `ReverseProxy:Clusters:api-cluster:Destinations:main-api:Address`,
but the chart used a fabricated destination name (`destination1`) and
transposed the cluster ID and destination name. As with #5, a mismatched
env var name is silently ignored by .NET configuration binding, so YARP
fell back to the `appsettings.json` default meant for local development.

**Fix:** Corrected the environment variable names to exactly match the
`ReverseProxy:Clusters:<clusterId>:Destinations:<destinationName>:Address`
path from the real `appsettings.json`, mirroring the pattern already used
correctly in the ACA Terraform configuration. General lesson: when
overriding nested configuration sections via environment variables, the
double-underscore path must match the JSON hierarchy exactly, including
which level holds the cluster ID versus the destination name.

### 8. Azure Load Balancer health probe silently dropped external traffic

**Symptom:** The Gateway was fully functional when tested from inside the
cluster (`kubectl run curl-test ...` against the internal Service), but the
public Load Balancer IP was completely unreachable from two independent
external networks (home WiFi and cellular data) — connections simply timed
out, with no trace of the requests in the ingress controller's access logs.

**Root cause:** AKS's cloud-provider integration configures the Azure Load
Balancer's health probe against the ingress controller's `NodePort` using
`GET /` by default. The Gateway has no route mapped to `/` (only `/api/*`
and `/api/identity/*`), so nginx-ingress returned `404` to the probe. Azure
interpreted the `404` as an unhealthy backend and stopped forwarding
external traffic to it entirely — even though the backend was, in every
practical sense, healthy. Because the health probe operates below the
application layer the request never reached nginx's access log, which is
why internal tests succeeded while external traffic vanished without a
trace.

**Fix:** Annotated the `ingress-nginx-controller` Service with
`service.beta.kubernetes.io/azure-load-balancer-health-probe-request-path=/healthz`,
pointing the probe at nginx-ingress's dedicated health endpoint (which
always returns `200`) instead of the application's own root path. General
lesson: a Kubernetes Service being healthy and routable internally does not
guarantee the cloud load balancer considers it healthy — always verify what
path/port the cloud provider's health probe actually checks, especially
when the application doesn't serve anything meaningful at `/`.

## Decision

Retain all of the above fixes as the canonical AKS deployment configuration
for TicketFlow. Document each root cause in this ADR rather than only fixing
symptoms, since several of these (env var name mismatches, LB health probe
defaults) are easy to reintroduce in future changes if the underlying
"why" isn't understood.

## Consequences

**Positive:**
- The AKS track now demonstrates genuine operational Kubernetes knowledge:
  Helm chart authoring, Secret management, Service networking, Ingress
  configuration, and cloud load balancer health probe tuning.
- Each of these issues is a realistic, interview-worthy debugging story,
  particularly #8 (the health probe issue), which is a well-known but
  frequently-missed AKS/nginx-ingress interaction.
- Reinforces the "each service owns its database" principle from the Phase
  5 Identity extraction, now enforced at the infrastructure level as well
  as the code level.

**Negative / trade-offs:**
- Running two Azure SQL databases (main + Identity) increases the demo's
  infrastructure footprint and cost slightly versus a single shared
  database, though this matches production-realistic microservices
  boundaries.
- The AKS track required substantially more manual configuration and
  debugging than ACA, reinforcing why ACA was chosen as the *primary*
  production deployment target (see ADR-016) — AKS demonstrates capability
  but ACA offers materially lower operational overhead for a workload of
  this size.
